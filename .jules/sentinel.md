## 2026-06-18 - Prevent Timing Attacks in OAuth Credentials
**Vulnerability:** OAuth `client.secret` was being checked using the standard `!==` operator in `worker.ts`. Because JavaScript's standard string equality operators fail fast (short-circuit on the first mismatched character), this exposed the application to timing attacks where an attacker could deduce secrets character by character by measuring response times.
**Learning:** Even though `constantTimeEqual` was already implemented in the codebase (for `bearerToken` comparison), it was not being consistently used for other sensitive secrets like `client.secret` during OAuth authorization grants. It is critical to use timing-safe comparisons for secrets.
**Prevention:** Always use a constant-time equality function (like `constantTimeEqual`) when comparing secrets, passwords, or authentication tokens. Make sure to validate that the secret exists before calling the function.

## 2024-06-22 - Prevent DoS and Timing Leaks in constantTimeEqual
**Vulnerability:** The custom `constantTimeEqual` function in `worker.ts` iterated up to the maximum length of the two strings being compared (`Math.max(leftBytes.length, rightBytes.length)`), substituting `undefined` with `0` for out-of-bounds accesses. This could allow an attacker to pass extremely long strings, causing a small Denial of Service (DoS) by forcing long iterations. Additionally, relying on out-of-bounds array access (`undefined`) can lead to V8 engine deoptimizations, potentially introducing new timing side channels.
**Learning:** For constant-time equality comparisons, it is safe and highly recommended to fail fast and return early if the lengths of the two inputs do not match. Since hashes, tokens, and secrets generally have known/fixed lengths, revealing a length mismatch via an early return does not provide an attacker with actionable information, but it prevents DoS and complex timing issues.
**Prevention:** Always implement an early return `if (leftBytes.length !== rightBytes.length) return false;` in custom `constantTimeEqual` implementations.
## 2026-06-25 - Prevent Timing Attacks in PKCE Challenge Validation
**Vulnerability:** The OAuth PKCE `code_challenge` was compared against the hashed `codeVerifier` using the standard `!==` operator in `worker.ts`. This standard string comparison short-circuits upon the first mismatched character, exposing the validation to timing attacks.
**Learning:** Cryptographic hashes and security tokens (like PKCE challenges) should always use timing-safe string comparison. The existing `constantTimeEqual` function is effective for mitigating this when evaluating tokens or hashed credentials.
**Prevention:** Use `constantTimeEqual` (or a similar timing-safe string comparison utility) instead of `!==` or `===` whenever comparing tokens, hashes, or secrets during authentication flows.
## 2026-06-25 - Prevent DoS via Input Length Limits
**Vulnerability:** The application was missing input length limits on the query parameter for MCP tools. This could allow an attacker to send excessively large queries, potentially leading to Denial of Service (DoS) or performance degradation by forcing the application to process massive payloads.
**Learning:** All user inputs, especially those passed to complex processing functions or external APIs, should have strict length boundaries to prevent resource exhaustion.
**Prevention:** Implement maximum length validation for all query inputs. Set reasonable limits based on the expected use case (e.g., 2000 characters for a search query) to block disproportionately large inputs early in the request lifecycle.
## 2026-06-25 - Prevent Brute-force on OAuth Endpoints via Rate Limiting
**Vulnerability:** The application applied its rate limiter only on the primary `/mcp` tool execution endpoint and ignored the OAuth `/oauth/token` and `/oauth/authorize` endpoints. This omission left the authentication endpoints exposed to brute-force attacks on OAuth authorization codes and potential Denial of Service (DoS) attacks on the worker.
**Learning:** Security mechanisms like rate limiting must cover all potentially sensitive POST endpoints, especially those dealing with authentication, token issuance, or authorization flows. Placing rate limit middleware too low in the routing structure bypasses protection for routes handled earlier in the file.
**Prevention:** In Cloudflare Worker HTTP request handlers, place the rate limiter logic early in the execution chain—after checking static/public unauthenticated assets (e.g., metadata, favicons) but *before* matching and processing OAuth routes.
